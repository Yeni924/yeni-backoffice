OPERATION IMPROVEMENT

운영 개선 경험과 포트폴리오 반영

기존 운영 업무에서 다뤘던 문제를 그대로 나열하기보다, 이번 포트폴리오에서 어떤 구조로 다시 풀어냈는지 중심으로 정리했습니다.

실무에서 다룬 문제 당시 상황 개선 또는 처리 방식 포트폴리오에 반영한 부분
외부 API 요청/응답 추적 PG, SAP, POS/KIOSK, 회원/포인트 API처럼 외부 시스템과 주고받는 데이터는 실패 시 원인 확인이 늦어지면 운영 대응이 어려웠습니다. 요청/응답 전문, 처리 결과, 실패 사유, 처리 시간을 분리해 확인할 수 있도록 로그 기준을 잡고 운영자가 추적할 수 있는 흐름을 중요하게 봤습니다. PG API 로그, 외부 전송 이력, requestId, elapsedMs, 실패 사유 컬럼을 두고 결제부터 외부 전송까지 같은 흐름으로 추적할 수 있게 구성했습니다.
알림톡 Agent 발송 처리 알림톡 발송은 단순 API 호출이 아니라, 발송 대상 조회, 발송 요청, 성공/실패 처리, 재시도 기준이 함께 필요한 운영성 작업이었습니다. DB Polling 방식으로 발송 대상을 가져오고, 대량 발송 시 병렬 처리 구조를 적용해 발송 처리량과 안정성을 함께 고려했습니다. 결제 승인, 취소, 외부 전송 실패 같은 이벤트가 발생하면 알림톡 Queue를 생성하고, Mock Agent가 Queue를 처리하는 구조로 연결할 예정입니다.
상품 마스터 동기화 성능 개선 상품 마스터 데이터를 POS/KIOSK 또는 테이블오더 쪽으로 동기화하는 과정에서 전체 데이터를 건별로 처리하면 시간이 오래 걸리고 운영 대기 시간이 길어졌습니다. 반복 update 흐름을 줄이고, 변경 대상 중심의 일괄 처리 방식으로 개선해 동기화 시간을 단축했습니다. 매출 통계와 외부 전송 재처리에서도 같은 관점으로 접근합니다. 운영 화면에서 매번 원장을 풀스캔하지 않도록 집계 테이블, View, 인덱스 적용을 검토합니다.
결제/취소 상태값 관리 결제는 승인 성공만 중요한 것이 아니라, 취소, 부분 취소, 중복 요청, 실패 후 재요청 같은 예외 흐름까지 같이 관리해야 했습니다. 원거래와 취소거래를 구분하고, 요청 중복 여부와 상태 변경 가능 여부를 기준으로 처리 흐름을 나누는 방식이 필요했습니다. 결제 승인 시 SALE 거래를 생성하고, 취소 성공 시 원매출을 수정하지 않고 CANCEL 음수 거래를 별도로 남기는 원장형 구조로 구현했습니다.
운영 관리자 화면 개선 운영자가 쓰는 화면은 단순 CRUD보다 상태 확인, 필터링, 재처리, 엑셀 다운로드, 실패 원인 확인 같은 기능이 실제 업무에 더 중요했습니다. 요청 의도를 확인한 뒤 화면 입력 흐름, 검증 조건, 조회 조건, 외부 시스템 반영 기준을 함께 맞춰 기능을 수정했습니다. PG Mock 콘솔에서 결제 상태, 매출 생성 여부, 외부 전송 상태, 실패 로그를 한 화면에서 확인할 수 있도록 구성했습니다.
SQL Injection 위험 구간 개선 동적 SQL 사용 구간에서 입력값이 SQL 문장에 직접 결합될 경우 보안 취약점과 유지보수 문제가 발생할 수 있었습니다. Dapper 파라미터 바인딩을 적용해 입력값과 SQL 문장을 분리하고, 반복되는 조회/수정 로직을 안전하게 정리했습니다. 포트폴리오에서는 JPA와 SQL Mapper를 역할에 따라 분리하고, 검색 조건과 통계 조회 조건은 파라미터 기반으로 처리하는 구조를 사용합니다.

02 완료
Mock PG 승인/취소 기본 흐름 구현

실제 PG 운영망과 연결하지 않고, Mock PG Client를 통해 승인/취소 성공 및 실패 흐름을 재현했습니다. 승인 성공 시 결제 거래와 매출 SALE 거래가 생성됩니다.

03 완료
매출 거래 원장 구조 구현

승인 건은 SALE, 취소 건은 CANCEL 음수 거래로 저장합니다. 원매출을 직접 수정하지 않고 거래 이력을 보존하는 방식으로 구성했습니다.

04 진행 중
외부 전송 실패/재처리 흐름 구현

매출 거래 생성 이후 외부 전송 대상을 만들고, Mock 전송 결과에 따라 성공/실패/재전송 상태를 관리하는 기능을 보강하고 있습니다.

05 진행 중
알림톡 Queue / Mock Agent 연결

결제 승인, 취소, 외부 전송 실패 같은 이벤트를 알림톡 Queue로 연결하고, Mock Agent가 Queue를 읽어 발송 성공/실패와 재시도 이력을 남기는 구조를 구현하고 있습니다.

06 진행 예정
매출/정산 통계 및 성능 테스트

SALE/CANCEL 거래 원장을 기준으로 매출 통계를 조회하고, 100만 건 테스트 데이터를 넣어 일반 조회, View, 집계 테이블 방식의 성능 차이를 비교할 예정입니다.

07 진행 예정
정산 배치 및 수수료 정책 확장

PG사/결제수단별 수수료 정책을 관리하고, 미정산 매출을 기준으로 DRAFT 정산 데이터를 생성하는 배치 흐름을 확장할 예정입니다.

08 진행 예정
테스트 코드 및 README 정리

결제 승인/취소, 매출 생성, 외부 전송 실패, 알림톡 발송 실패, 정산 배치 흐름을 테스트 코드와 README로 정리할 예정입니다.
