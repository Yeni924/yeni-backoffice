실무 경험

실무에서 했던 일

실제 담당했던 업무를 기준으로, 어떤 상황에서 어떻게 처리했는지 정리했습니다.

업무 배경

PG, 배송, 회원, 포인트, POS/KIOSK 관련 API처럼 외부 시스템과 연결되는 기능은 실패가 발생했을 때 요청값과 응답값, 실패 사유를 빠르게 확인하는 게 중요했습니다.

실무 인사이트

운영 중 문제가 생겼을 때 원인을 찾을 수 있도록 요청/응답 데이터, 처리 결과, 실패 메시지, 처리 시간을 남기는 구조가 필요하다고 느꼈습니다.

포트폴리오 반영

포트폴리오에서는 PG API 로그, 외부전송 이력, requestId, elapsedMs, 실패 사유를 분리해 저장하고 결제 승인부터 외부전송까지 같은 흐름으로 추적할 수 있도록 구성했습니다.

업무 배경

알림톡 발송은 단순 API 호출로 끝나는 기능이 아니라 발송 대상 조회, 발송 요청, 성공/실패 처리, 재시도 기준까지 함께 고려해야 했습니다.

실무 인사이트

DB Polling 방식으로 발송 대상을 조회하고, 대량 발송 시 병렬 처리 구조를 적용해 발송 성능을 개선했습니다.

포트폴리오 반영

포트폴리오에서는 결제 승인, 취소, 외부전송 실패 같은 이벤트가 발생하면 알림톡 Queue를 생성하고, 실제 발송 처리는 결제 흐름과 분리할 수 있도록 구조를 잡았습니다.

업무 배경

상품, 매장, 카테고리, 옵션 같은 운영 데이터는 관리자 화면에서 등록·수정되고, POS/KIOSK 또는 외부 시스템에서 사용할 수 있는 형태로 제공되어야 했습니다.

실무 인사이트

단말 담당자나 연동 담당자와 필요한 필드, 상태값, 조회 조건, 반영 기준을 직접 확인하며 관리자 화면과 API 응답 구조를 맞춰가는 작업을 반복했습니다.

포트폴리오 반영

포트폴리오에서는 외부 시스템이 필요로 하는 데이터 기준을 명확히 두고, 요청/응답, 상태값, 로그를 분리해 확인할 수 있는 백오피스 구조로 정리했습니다.

업무 배경

결제는 승인 성공만 관리하면 되는 것이 아니라, 취소, 부분취소, 중복 요청, 실패 후 재요청 같은 예외 흐름까지 함께 고려해야 했습니다.

실무 인사이트

원거래와 취소거래를 구분하고, 요청 중복 여부와 상태 변경 가능 여부를 기준으로 흐름을 나눠야 했습니다.

포트폴리오 반영

포트폴리오에서는 결제 승인 시 SALE 거래를 생성하고, 취소 성공 시 원매출을 수정하지 않고 CANCEL 음수 거래를 별도로 남기는 원장형 구조로 구현했습니다.

업무 배경

운영자가 실제로 필요로 하는 기능은 단순 등록·수정보다 상태 확인, 조건 검색, 실패 원인 확인, 재처리 여부 판단에 가까웠습니다.

실무 인사이트

요청 의도를 먼저 파악한 뒤, 화면 입력 흐름, 검증 조건, 조회 조건, 외부 시스템 반영 기준을 함께 맞춰가며 기능을 수정했습니다.

포트폴리오 반영

포트폴리오에서는 PG 운영 콘솔에서 결제 상태, 매출 원장 생성 여부, 외부전송 상태, 알림톡 Queue, RecoveryTask를 한 화면에서 확인할 수 있도록 구성했습니다.

업무 배경

운영성 화면에서는 검색 조건이 많아지면서 동적 SQL이나 조건 분기 로직이 복잡해지는 경우가 있었습니다. 입력값 처리 방식이 명확하지 않으면 유지보수와 안정성 양쪽에서 문제가 생길 수 있었습니다.

실무 인사이트

Dapper 사용 시 파라미터 바인딩을 적용해 입력값과 SQL 문장을 분리하고, 반복되는 조회·수정 로직을 정리하는 방향으로 개선했습니다.

포트폴리오 반영

포트폴리오에서는 JPA Repository와 조건 기반 조회를 사용해 검색 조건을 파라미터로 전달하고, 매출 원장 조회는 DB-side paging과 집계 쿼리로 처리했습니다.

구현 범위

포트폴리오 구현 범위 및 진행 상태

지금 바로 확인할 수 있는 부분, 구현 중인 부분, 이후 확장 예정인 부분을 나눠서 정리했습니다.

구분 상태 현재 구현 범위 실무 경험 화면/기술적으로 보여줄 부분
Mock PG 결제 승인 시연 가능 실제 PG 운영망과 직접 통신하지 않고, Mock PG Client를 통해 결제 승인 성공/실패 흐름을 재현합니다. 결제 요청 후 승인 결과를 저장하고, 성공 건은 결제 거래와 SALE 매출 원장으로 연결합니다. 티쿤글로벌에서 Stripe/Paymentwall 연동 업무를 담당했습니다. 결제 승인 이후 주문 상태, 결제 상태, 후속 처리 흐름을 함께 고려해야 했습니다. 결제 상태값, 승인 결과 저장, idempotencyKey 기반 중복 요청 방지, PG 요청/응답 로그 저장
Mock PG 결제 취소 시연 가능 승인된 결제 건에 대해 전체 취소 또는 부분 취소를 요청할 수 있습니다. 취소 성공 시 취소 이력을 저장하고, 매출 원장에는 CANCEL 음수 거래를 별도로 생성합니다. 결제/취소 흐름에서 원거래와 취소거래를 구분하고, 중복 요청이나 부분취소 금액 초과 같은 예외를 함께 고려했습니다. 누적 취소금액 검증, 중복 취소 방지, 부분취소 처리, CANCEL 음수 거래 생성, 원 SALE 거래 연결
매출 거래 원장 시연 가능 결제 승인 성공 시 SALE 거래를 생성하고, 결제 취소 성공 시 원매출을 수정하지 않고 CANCEL 음수 거래를 별도로 남깁니다. 통계와 정산은 이 거래 원장을 기준으로 계산합니다. POS/KIOSK 및 영업관리 업무에서 매출 데이터, 결제 정보, 외부 전송 상태를 함께 다뤘습니다. 매출은 단순 합계가 아니라 원거래, 취소, 전송 여부를 추적할 수 있어야 했습니다. SALE/CANCEL 거래 구분, 원거래 연결, BigDecimal 금액 처리, sourceType/sourceId 기반 중복 원장 방지
PG API 로그 시연 가능 결제 요청, 승인 응답, 취소 요청, 취소 응답을 로그로 남깁니다. 민감정보를 제외한 Mock 요청/응답 기준으로 저장하고, 실패 원인을 추적할 수 있게 구성했습니다. 외부 API 연동 장애가 발생했을 때 요청/응답 로그와 처리 시간이 없으면 원인 파악이 늦어지는 상황을 경험했습니다. pg_api_log, requestId, elapsedMs, resultCode, resultMessage, 실패 사유 추적
외부 전송 관리 시연 가능 매출 거래가 생성되면 외부 시스템으로 전달해야 할 전송 대상을 생성합니다. 실제 외부 운영망과 직접 연동하지 않고, Mock 전송 상태와 실패 사유를 관리합니다. PG, 배송, 회원, 포인트, POS/KIOSK 등 외부 시스템과 연결되는 기능을 다루면서 실패 건을 놓치지 않고 추적하는 구조의 중요성을 직접 경험했습니다. READY/SENDING/SENT/FAILED/RETRY_READY 상태관리, 전송 이력 저장, 실패 사유, 재전송 횟수 제한
알림톡 Queue 시연 가능 결제 승인, 결제 취소, 외부전송 실패 같은 이벤트가 발생하면 알림톡 발송 Queue를 생성합니다. 실제 알림톡 API 호출은 하지 않고, 발송 대상과 상태를 분리해 관리합니다. DB Polling 방식으로 발송 대상을 조회하고, 대량 발송 시 병렬 처리 구조를 적용해 발송 성능을 개선한 경험이 있습니다. alimtalk_queue, messageKey 중복 방지, READY/SENDING/SUCCESS/FAILED/RETRY 상태, 결제 흐름과 발송 흐름 분리
RecoveryTask 시연 가능 승인 결과불명, 취소 결과불명, 망취소 필요, 후속처리 실패처럼 운영자가 확인해야 하는 건을 RecoveryTask로 남깁니다. 목록/상세/재처리 API를 통해 복구 흐름을 추적할 수 있습니다. 외부 연동이 포함된 업무에서는 실패를 단순 오류로 끝내지 않고, 어떤 요청이 실패했는지 다시 확인하고 재처리할 수 있는 구조가 필요했습니다. READY/PROCESSING/SUCCESS/FAILED 상태, orderNo, tid, requestId 추적, retry API, 실패 사유 관리
API 로그 / 운영 로그 시연 가능 PG 로그, 외부전송 로그, 알림톡 Queue, 관리자 처리 로그를 분리해 어떤 단계에서 실패했는지 확인할 수 있도록 구성했습니다. 운영 이슈가 생겼을 때 원인을 찾으려면 로그 기준이 명확해야 했고, 담당자 간 원인을 공유할 때도 요청 단위 추적 정보가 중요했습니다. requestId, MDC, Global Exception, audit_log, external_send_history, settlement_log
매출/정산 요약 시연 가능 SALE, CANCEL 거래를 기준으로 총 SALE 금액, 총 CANCEL 금액, 순매출, 정산 상태별 건수를 조회합니다. 요약은 현재 페이지가 아니라 조회 조건 전체를 기준으로 계산합니다. Dapper 기반으로 매출 조회 및 집계 쿼리를 작성한 경험이 있습니다. 운영 화면에서는 조건 검색, 기간 조회, 상태별 집계가 중요했습니다. DB-side paging, summary 집계 쿼리, 기간 조건 조회, 상태별 집계, 인덱스 고려
PG 수수료 정책 기본 구조 구현 PG사, MID, 결제수단별 수수료율과 적용 기간을 관리하는 구조를 둡니다. 정산 계산 시점의 수수료 정책을 조회해 수수료와 수수료 VAT를 계산합니다. 정산 정책 전체를 직접 운영한 경험은 아니지만, 결제와 매출 데이터가 정산 계산으로 이어지려면 수수료 정책이 분리되어야 한다고 판단해 설계했습니다. feeRate, fixedFee, 적용 기간, BigDecimal 계산, 정책 기간 중복 검증, 수수료 계산 결과 저장
정산 배치 시연 가능 미정산 SALE/CANCEL 원장을 기준으로 정산 데이터를 생성하고, 운영자가 확인 후 확정/지급 완료 처리할 수 있는 흐름을 구현했습니다. 동일 기간/MID 기준 중복 정산도 방지합니다. 정산 시스템을 직접 운영한 경험보다는, 결제·매출·외부전송 흐름을 다루며 정산으로 이어질 데이터 기준을 이해한 경험을 바탕으로 구성했습니다. settlement_statement, settlement_detail, DRAFT/CONFIRMED/PAID 상태관리, 확정 후 재계산 제한, 지급 완료 후 수정 제한
상품/매장·카테고리 관리 경력 설명 중심 이 포트폴리오의 핵심은 결제 이후 매출·전송·알림·정산 흐름이라 상품/매장/카테고리 관리 화면은 구현 범위에서 분리했습니다. KFC, 패키지 키오스크, 땡겨요 관련 운영 업무에서 상품, 매장, 카테고리, 옵션 등 운영 마스터 데이터를 관리자 화면과 API 기준으로 관리하는 업무를 직접 담당했습니다. 후속 확장 시 상품 마스터, 매장별 사용 여부, 카테고리, 옵션, POS/KIOSK 조회 API 기준으로 확장 가능
개발 흐름

개발 진행 흐름

결제 승인/취소에서 매출 원장, 외부전송, 알림톡 Queue, 복구 작업, 정산 기본 흐름까지 공개 데모에서 확인할 수 있는 순서로 정리했습니다.

01 완료
프로젝트 기본 구조 및 경력 대시보드 구성

Spring Boot 기반 백오피스 화면 구조, 프로젝트 목록, 상세 모달, DB 명세 화면, Swagger API 문서 연결을 구성했습니다.

02 완료
Mock PG 승인/취소 기본 흐름 구현

실제 PG 운영망과 연결하지 않고, Mock PG Client를 통해 승인/취소 성공 및 실패 흐름을 재현했습니다. 승인 성공 시 결제 거래와 매출 SALE 거래가 생성됩니다.

03 완료
매출 거래 원장 구조 구현

승인 건은 SALE, 취소 건은 CANCEL 음수 거래로 저장합니다. 원매출을 직접 수정하지 않고 거래 이력을 보존하는 방식입니다.

04 완료
외부전송 대기/실패 흐름 구현

매출 거래 생성 이후 외부전송 대상을 만들고, Mock 전송 상태와 실패 사유를 별도로 관리하는 구조를 구현했습니다.

05 완료
알림톡 Queue 생성 구조 구현

결제 승인, 취소, 외부전송 실패 이벤트를 알림톡 Queue로 연결하고, 수동 실행 Mock Worker로 성공·실패·재시도 흐름을 확인할 수 있도록 구성했습니다.

06 완료
RecoveryTask 조회/재처리 API 구현

승인 결과불명, 취소 결과불명, 망취소 필요, 후속처리 실패 건을 RecoveryTask로 추적하고 목록/상세/재처리 API에서 확인할 수 있게 구성했습니다.

07 완료
매출 원장 조회와 요약 집계 구현

SALE/CANCEL 원장을 DB-side paging으로 조회하고, 조회 조건 전체 기준의 summary 집계를 제공하도록 구성했습니다.

08 완료
정산 배치 기본 흐름 구현

미정산 SALE/CANCEL 원장을 기준으로 정산 데이터를 생성하고, 확정/지급 완료 상태로 이어지는 기본 흐름을 구현했습니다.

09 보강 예정
자동 Worker와 대량 데이터 검증 확장

실제 외부망 연동, 자동 스케줄링, dead-letter 처리, 입점몰 정산 확장, 대량 데이터 성능 테스트는 후속 보강 범위입니다.
